handling errors

clientcord provides structured error handling. each type of error can be caught separately on your command class.

error in command execution

if an error is thrown in the run method, clientcord calls on_error.

commands/fail.py
from clientcord.declare import command

@command(name="fail", description="a command that fails")
class FailCommand:
    async def run(self, ctx):
        raise Exception("something broke")

    async def on_error(self, ctx, error):
        await ctx.write(f"error: {error}")

error in option validation

when an option fails validation, clientcord calls on_option_error with a list of error strings.

commands/validated.py
from clientcord.declare import command, options
from clientcord.commands.options import string_option

@options(
    url=string_option(description="a valid url", required=True)
)
@command(name="fetch", description="fetch a url")
class FetchCommand:
    async def run(self, ctx, url: str):
        await ctx.write(f"fetching {url}")

    async def on_option_error(self, ctx, errors):
        await ctx.write(f"invalid options: {', '.join(errors)}")

error from middleware

when a middleware calls stop(), clientcord calls on_middleware_error.

commands/restricted.py
from clientcord.declare import command, middleware

@middleware("owner_only")
async def owner_check(ctx, next_mw, stop):
    if ctx.user.id != "owner-id":
        return stop("not the bot owner")
    await next_mw()

@command(name="restart", description="restart the bot")
@owner_check
class RestartCommand:
    async def run(self, ctx):
        await ctx.write("restarting")

    async def on_middleware_error(self, ctx, error):
        await ctx.write(f"blocked: {error}")
if you don't define these error handlers on your command class, clientcord uses default handlers that log the error and reply with a generic message.
